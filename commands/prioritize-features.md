# 기능 우선순위 판단

한정된 리소스에서 무엇을 먼저 해야 하는가를 이슈 드리븐 관점으로 판단합니다.

## 실행 절차

1. **판단 대상 명확화**
   - `$ARGUMENTS` 가 비어 있으면 사용자에게 한 번에 다음 3가지를 질문:
     - 판단 대상 기능/과제 목록
     - 가용 리소스 (인원, 기간)
     - 달성 목표 (사업/기술 KPI)

2. **현황 분석**
   - 코드베이스에서 관련 영역 직독
   - 이미 부분 구현된 부분 / 완전 신규 개발 구분
   - 의존성(다른 기능의 선행 조건 여부) 파악

3. **consultant-kr 에이전트 호출**
   - 적용 프레임워크:
     - **이슈 드리븐**: 각 과제의 "이슈도" 평가
     - **파레토(80/20)**: 20% 기능이 80% 가치를 만드는 후보 식별
     - **RICE 스코어** (Reach × Impact × Confidence ÷ Effort) 병용
   - 출력 요구사항:
     - 각 기능별 평가 테이블
     - 선정된 기능과 탈락 기능의 명확한 구분
     - 선정 이유 (So What? / Why So? 포함)
     - 탈락한 기능도 추후 재검토 시점 명시

## 출력 형식

```markdown
## 판단 결과 요약

**선정**: 기능 A, 기능 C
**보류**: 기능 B (→ 다음 분기 재검토)
**탈락**: 기능 D, 기능 E

## 평가 테이블

| 기능 | Reach | Impact | Confidence | Effort | RICE | 이슈도 | 결정 |
| --- | :-: | :-: | :-: | :-: | :-: | :-: | :-: |
| A | 8 | 9 | 0.9 | 3 | 21.6 | 상 | 선정 |
| B | 5 | 6 | 0.7 | 5 | 4.2 | 중 | 보류 |
| ... |

## 선정 이유 (공·우·산)

### 공 (사실)
- 리소스: ...
- 현재 구현도: ...

### 우 (해석)
- 파레토 관점: 상위 X개가 전체 가치의 Y% 차지
- 이슈도 관점: 핵심 논점(KPI 달성)과의 연결성

### 산 (제안)
- 선정 기능의 실행 로드맵
- 보류 기능의 재검토 트리거 조건
```

## 중요 규칙

- "전부 중요합니다" 같은 답변 **절대 금지** (컨설턴트 정의에 명시됨)
- 선정 기준을 정량화해야 함 (감으로 판단 금지)
- 탈락 기능에도 명확한 이유 제시 (의사결정 추적 가능성 확보)
- 선정 개수는 **사용자가 지정한 개수를 반드시 준수**

## 한국어 윤문 (humanize-korean 스킬 활용 — 옵트인)

위 결과를 사용자에게 출력한 직후, 다음 경로 중 하나에 `SKILL.md` 가 존재하는지 확인한다:
- `~/.claude/skills/humanize-korean/SKILL.md`
- `./.claude/skills/humanize-korean/SKILL.md`

**존재하면**: `humanize-korean` 스킬을 호출하여 위 결과의 한국어 서술 부분에만 윤문을 적용한 뒤, "🪶 한국어 윤문본" 섹션으로 추가 출력한다. 원본 결과는 그대로 유지.

**존재하지 않으면**: 단계 건너뛰고 마지막에 한 줄 안내만:
- `💡 한국어 윤문 추가: npx consultant-kr-cli humanize --local`

**윤문 시 절대 변경 금지**: 기능명, RICE 점수와 평가 테이블의 모든 수치, 결정값(선정/보류/탈락), Phase 번호. 자연어 설명 문장만 다듬는다.
